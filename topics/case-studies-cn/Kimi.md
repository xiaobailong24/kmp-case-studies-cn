# 背景

随着 Kimi 用户群体的持续壮大，其使用场景愈发多元化，涵盖了多种设备类型。为了满足用户在不同设备上便捷、高效地使用 Kimi 的需求，Kimi 客户端开发团队致力于打造支持多端的客户端应用。在技术选型上，团队紧跟行业潮流，积极探索并实践了基于 KMP（Kotlin Multiplatform）+CMP（Compose Multiplatform）的跨端开发方案。
传统的多端开发模式通常是针对每个平台单独开发一套代码，这种方式在开发周期、功能一致性、代码维护等方面存在诸多困难。鉴于此，我们决定探索跨端开发方案，期望找到一个具备原生性能、优异互操作性以及现代化语言支持的框架。经过深入调研与评估，我们最终选择了 KMP 作为核心框架，以实现跨平台的高效开发。

# 集成和实现

## 历程

- 2024 年 6 月：启动已有 Android 工程的改造工作，目标是实现基础逻辑和 UI 在 PC（Windows 和 Mac）以及 Android 平台上的共享。
- 2024 年 10 月：基于 CMP 的 Android 改造完成，并进行了小规模测试。同时，团队着手开展针对纯鸿蒙（OHOS）的跨端调研与实践工作。
- 2024 年 11 月：成功上线基于 CMP 的 Android 和 PC 版本，为用户提供了跨平台的使用体验。
- 2024 年 12 月：发布第一版基于 KMP 的鸿蒙原生 Kimi 应用，进一步拓展了跨端覆盖范围。

## 跨端开发方案概述

1. 鸿蒙（通过 Kotlin/JS 实现）、PC 和 Android 逻辑共享层：该层代码实现了跨三端共享，即同时适用于鸿蒙（通过 Kotlin/JS 实现）、PC 和 Android 三个平台。Share 层主要负责处理业务逻辑、数据模型定义、网络请求、日志等与界面无关的核心功能。通过将这些核心功能集中到 Share 层，有效避免了在不同平台重复编写相似代码，提高了代码复用性和开发效率。
2. PC 和 Android 逻辑共享层：该层专门用于共享仅适用于 PC 和 Android 的逻辑，主要负责数据库操作、图片加载等的统一实现，进一步优化了跨平台开发流程。
3. PC 和 Android UI 共享层：我们积极探索并实践了 MVI（Model-View-Intent）相关架构的应用，实现了跨两层共享。PC 和 Android 端共享一套 UI 代码，而鸿蒙系统则使用原生 ArkTS 进行 UI 实现，充分发挥各平台的优势，同时保持了用户体验的一致性。

## PC 开发中的实践经验

1. KV 实践：在 PC 端开发中，我们借鉴 Android 的开发经验，使用 AndroidX 的 Data Store 来实现与 Android 端对等的 KV（Key-Value）存储功能，确保数据存储的一致性和便捷性。
2. 数据库实践：Kimi 应用中使用数据库存储数据，我们选择了 Room 数据库来实现数据的存储，借助其强大的功能和良好的兼容性，保障了数据的高效存储与访问。
3. Mac OS 系统托盘（Trayicon）的深浅适配：在 CMP 开发中，为了使应用在 macOS 系统中更好地适配深色和浅色主题，我们设置了 Java 的环境参数 apple.awt.enableTemplateImages。通过这一设置，让关联的所有图像被 macOS 桌面系统视为模板图像，系统会根据当前的桌面主题和颜色自动调整图像，使其在不同背景下保持清晰可见，提升了应用在 macOS 系统中的视觉体验。
4. 适配 Windows 下的焦点变化导致的屏幕闪烁问题：在 Windows 平台上，CMP 存在焦点闪烁问题，影响用户体验。经过探索，我们发现将 Skiko 的渲染后端设置为 OpenGL（skiko.renderApi = opengl）可以有效解决这一问题，使 Skiko 能够稳定渲染，避免屏幕闪烁，确保应用在 Windows 系统中的流畅运行。
5. 音频播放器：在 CMP 中，目前缺乏特别好用的开箱即用的音频播放器方案。我们尝试过使用 Java 原生的 javax audio player，但效果并不理想，存在起播慢、无法设置播放列表等问题。经过深入探索，我们发现 VLC 播放器可以完美支持音频播放需求，尽管官方并不直接支持 CMP，但开源社区提供了基于 Java 的 API 封装层，我们通过直接调用 VLC 的动态库来实现音频播放功能，为用户提供了更优质的音频体验。
6. 其他功能探索：我们还相继探索了多窗口、热键、打点 SDK、异常上报等常用功能的实现，不断丰富和完善 PC 端应用的功能特性，提升应用的实用性和用户体验。

# 跨端面临的挑战

## PC 中的挑战

1. 与原生 Window 的交互问题：基于 Java AWT 的交互方式无法方便地调用一些高级功能，限制了应用在 Windows 和 mac 平台上的功能拓展。为了解决这一问题，我们积极探索基于 JetBrains 的 Java Runtime，以优化交互和 UI 实现，提升应用在 Windows 和 mac 系统中的表现。
2. 打包工具的限制：在 Windows 平台下，打包工具的选择受限于 jpackage 和 wixtoolset，难以满足我们对 Windows 安装包的定制化要求。因此，我们同步探索了基于 InnoSetup 的自定义打包工具，以实现更符合项目需求的安装包制作，提高应用的分发效率和用户体验。

## Kotlin JS 在鸿蒙中的挑战

1.  三方库适配问题：一些三方库的 JS 实现（如 Ktor）默认在 Node 或 Browser 环境中运行，无法直接适用于鸿蒙系统。为了克服这一难题，我们通过自定义 Ktor 的 OHOS client 来对接鸿蒙原生网络框架，确保网络请求等功能在鸿蒙系统中的正常运行，保障了跨平台开发的顺利进行。
2.  性能与多线程问题：在性能优化和多线程处理方面，我们仍在积极探索更加合适的解决方案，如 K/N（Kotlin Native）等技术的应用，以进一步提升应用在鸿蒙系统中的性能表现和稳定性，为用户提供更流畅的使用体验。

通过以上实践与探索，Kimi 客户端开发团队在跨端开发领域取得了显著进展，积累了丰富的经验。未来，我们将继续优化跨端开发方案，攻克更多技术难题，为用户提供更优质、更便捷的跨平台使用体验。
